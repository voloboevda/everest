# Product Logic

## Core question

Every UI element must answer:

> What does the user gain from this?

## Remove rule

If an element does not:

- increase understanding, OR
- improve conversion, OR
- reduce friction

→ **Remove it.**

## Simplicity

Before adding any element ask: Does it improve clarity, conversion, or usability?

If not → do not add it. Prefer less UI. Remove visual noise.

## Content first

Design exists to support content. Never create UI without understanding who reads, what they need, what action they should take.

## Everest primary metric

**Clarity > everything.**

Single RFQ funnel. B2B partner trust. No feature overload.

## Conflict resolution

If product logic conflicts with aesthetics → product logic wins.

If product logic conflicts with novelty → consistency wins.

## Production skeptic

Always assume the user is distracted and scanning. Optimize for fastest path to RFQ (`#request`).

## Memory

Decisions that embody product logic are logged in `docs/agents-log.md` and `docs/design-taste.md`. Preserve them across agent sessions.

Example decisions already made:

- Single Hero CTA (not duplicate per section)
- No theme toggle
- Static hero image (no video/beam)
- Equal-weight geography (no RU→CN emphasis)
- 4 sector cards (removed non-commodity categories)

Do not reconsider these without explicit user request.
